# テスト戦略

## 概要

投票システムの信頼性はテストカバレッジに直結する。Phase 1から以下の3層テストを実施する。

## テストフレームワーク

- **ユニットテスト / 統合テスト**: Vitest（Next.js + TypeScriptとの相性が良い）
- **E2Eテスト**: Playwright（ブラウザ内暗号化の検証に必要）
- **カバレッジ**: c8（Vitestに内蔵）

## 1. ユニットテスト

### 暗号化・復号（lib/crypto.ts）
- 鍵ペア生成 → 暗号化 → 復号で元の候補者IDが復元されること
- 異なるノンスで同じ候補者を暗号化しても異なる暗号文になること
- 不正な秘密鍵での復号が失敗すること
- 不正な暗号文の復号が失敗すること
- Ballot Trackerが同じ暗号文に対して同じハッシュを返すこと

### 住所パーサー（lib/district.ts）
- `docs/address-parser.md` のテストケース表をすべてカバー
- 基本パターン（市、区、政令指定都市の区）
- エッジケース（ひらがな市名、郡部、県名=市名）
- 異常系（都道府県なし、空文字列）

### 選挙区判定（lib/district.ts）
- 松戸市の住所 → 千葉6区
- 千代田区の住所 → 東京1区
- 選挙区マスターに存在しない市区町村 → null
- districtIdがnullの選挙（住所制限なし）→ 常に資格あり

### HMAC・ハッシュチェーン（lib/integrity.ts）
- HMAC生成と再検証が一致すること
- タイムスタンプが異なるとHMACが不一致になること
- ハッシュチェーンの正常検証が成功すること
- ハッシュチェーンの途中改ざんで検証が失敗すること
- ハッシュチェーンのBallot削除で検証が失敗すること
- ハッシュチェーンのBallot順序入替で検証が失敗すること

### CSRF（lib/csrf.ts）
- トークン生成が64文字の16進文字列であること
- 正しいトークンの検証が成功すること
- 異なるトークンの検証が失敗すること
- 空文字列・null・undefinedの検証が失敗すること

### 入力検証（lib/validation.ts）
- 正常なリクエストがバリデーションを通過すること
- 不正なcuid、Base64以外の文字列、サイズ超過で拒否されること
- 選挙作成スキーマの各フィールドの境界値テスト

## 2. 統合テスト（API Routes）

### 認証フロー（/api/auth/*）
- モック認証で正常にセッションが作成されること
- セッションにsubjectHashとdistrictのみが保存されていること（氏名・住所がないこと）
- 認証後15分でセッションが無効になること
- CSRFトークンがセッションに含まれていること

### 投票API（/api/vote）
- 認証済み + 資格あり + 有効な暗号データで投票が成功すること
- 未認証で401が返ること
- 選挙区外で403が返ること
- 投票期間外で400が返ること
- CSRFトークン不一致で403が返ること
- 不正なBase64で400が返ること
- レート制限超過で429が返ること
- 再投票時に旧BallotのisLatest=falseになること
- 再投票時にVoterRegistryのlatestBallotIdが更新されること
- 投票レスポンスにBallot Trackerが含まれること
- エラーレスポンスにスタックトレースが含まれないこと

### 開票API（/api/elections/[id]/count）
- ハッシュチェーン検証 → HMAC検証 → isLatest=trueのみ復号 → 集計が正しいこと
- 有効投票数とVoterRegistryのhasVoted=true件数が一致すること
- 選挙ステータスがOPEN以外で開票開始できないこと（CLOSEDにしてからCOUNTING）
- 不正な秘密鍵で復号が失敗しエラーが返ること

### 選挙管理API（/api/admin/elections）
- 選挙の作成・更新・ステータス遷移が正しいこと
- 候補者の追加・更新が正しいこと
- 管理者認証なしで401が返ること

## 3. E2Eテスト（Playwright）

### 投票フロー全体
1. 選挙トップページの表示
2. モック認証（松戸市在住ユーザーを選択）
3. 選挙区判定で資格ありの確認
4. 候補者一覧の表示（ランダム順）
5. 候補者を選択 → 確認画面
6. 「暗号化して投票する」ボタン → ブラウザ内で暗号化が実行されること
7. 投票完了画面にBallot Trackerが表示されること

### 選挙区外ユーザーの拒否
1. モック認証（千代田区在住ユーザーを選択）
2. 千葉6区の選挙に対して「投票資格がありません」が表示されること

### 再投票フロー
1. 初回投票を完了
2. 同じユーザーで再度投票
3. 新しいBallot Trackerが発行されること
4. 開票後、最後の投票のみが集計に反映されること

## 4. セキュリティテスト（自動化）

### CIパイプラインで毎回実行
- `npm audit` でcritical/highが0件であること
- CSPヘッダーが正しく設定されていること（レスポンスヘッダーの検証）
- APIルートがCSRFトークンなしのPOSTを拒否すること
- APIルートが不正な入力をすべて拒否すること

### 手動実行（Phase 3）
- OWASP ZAPによるスキャン
- 負荷テスト（k6 or Artillery）

## テストの実行方法

```bash
# ユニットテスト + 統合テスト
npm run test

# カバレッジ付き
npm run test:coverage

# E2Eテスト（開発サーバーが起動していること）
npm run test:e2e

# セキュリティ監査
npm audit --omit=dev
```
